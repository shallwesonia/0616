# 机器狗 MQTT 接口调整

## 任务背景

用户提供两份机器狗 MQTT command/result 终版协议文档，要求参考附件对整体接口进行调整，附件中已有内容尽量按附件格式落地。

## 用户需求

- MQTT command 按附件格式统一到 `factory/dogs/{robotCode}/command`。
- MQTT result 按附件格式统一到 `factory/dogs/{robotCode}/result`。
- command payload 使用 `schemaVersion/messageType/commandId/taskId/requestId/robotCode/traceId/command/issuedAt/timeoutMs/operatorId/source/params`。
- result payload 使用 `schemaVersion/messageType/event/eventId/commandId/taskId/requestId/robotCode/traceId/source/timestamp/data/error`。
- 同步调整 API 契约、MQTT 契约、虚拟执行体、链路追踪和文档。

## 当前代码现状

- 当前 MQTT Topic 为 `sim/{env}/{siteId}/robot/{robotId}/{messageType}`。
- 当前 payload 使用平台自定义 envelope，并将业务数据放在 `payload` 内层。
- 虚拟执行体分别发布 `ack/state/event/heartbeat/capability` 多类 Topic。
- 前端通过 REST API 创建 command，不直接连接 MQTT。

## 风险点

- Topic 和 payload 改动会影响已有 MQTT 联调脚本。
- 旧 `ack/state/heartbeat` 逻辑需要映射到 result event。
- 前端仍使用 `robotId/commandType`，需要 REST 层兼容转换。
- 历史 `data/state.json` 中已有旧格式消息，链路追踪和状态读取需要兼容。
