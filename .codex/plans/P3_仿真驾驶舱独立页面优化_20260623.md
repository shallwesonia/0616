# P3 仿真驾驶舱独立页面优化

## 任务背景

当前平台已经完成 P0/P1/P2 核心基线：

- P0：二维地图编辑、坐标轴、鼠标坐标、地图对象编辑、WebSocket 运行态展示、控制台事件触发、基础健康检查已完成。
- P1：PostgreSQL 主存储、Alembic migration、Redis 运行态缓存、数据库备份恢复、Docker Compose 部署闭环已完成。
- P2：MQTT command/result 契约、机器狗 MQTT 接口适配、消息总成、消息查询、trace 查询、局域网暴露和连接标准化已完成。

下一阶段正式进入 P3：统一运行模型与仿真驾驶舱。P3 的核心不是继续强化地图编辑，而是建立独立的 `/simulation` 仿真驾驶舱页面，让用户在已配置好的二维场地、机器人、动作集和任务流基础上，完成一次可追踪、可诊断、可导出的仿真运行。

## 用户需求

本阶段需要优化仿真驾驶舱页面规划，不进行代码编写。仿真驾驶舱必须作为单独页面建设，页面核心目标完成 6 件事：

1. 选场景：选择已经配置好的二维场地、机器人、动作集、任务流。
2. 建任务：手动创建一条模拟任务，或者从任务模板生成任务。
3. 发指令：通过消息总成下发动作指令，不直接操作执行体。
4. 看状态：查看机器人位置、状态、任务进度、资源占用。
5. 看消息：查看 Command、Telemetry、Event、Ack、Alert 等消息流。
6. 注入异常：模拟机器人离线、动作失败、路径阻塞、接口超时、消息丢失等异常。

## 风险点

- 运行模型风险：如果直接从页面下发设备命令，会破坏“消息总成统一下发、执行体只接 MQTT”的架构边界。
- 数据一致性风险：`MessageRecord`、`Observation`、`CurrentState` 和 `Trace` 必须能通过 `traceId/commandId/taskId/runId` 串联，否则后续无法解释任务成败。
- 页面职责风险：仿真驾驶舱不应变成第二个地图编辑器，场地只能选择和只读展示，正式地图编辑仍在地图页面完成。
- 异常注入风险：异常不能直接篡改机器人最终状态，必须先进入事件或 Observation，再由状态聚合逻辑影响 CurrentState。
- 规模风险：P3 先按单 Workspace 完成闭环，P5 再做 1-20 Workspace 的控制平面和实例隔离。
- 真实机器人风险：异常注入、消息丢失、动作失败等能力在接真实设备前必须默认隔离到仿真执行体或测试 Workspace。

## 待确认问题

默认采用以下假设，不作为当前阶段阻塞项：

- P3 先做单 Workspace 仿真驾驶舱，沿用当前 `workspace-local` 或现有默认 Workspace。
- 任务模板先使用平台内置和数据库配置，不在 P3 首轮实现复杂模板设计器。
- Agent 在 P3 首轮可以先使用规则式 Plan 生成，AI Agent 接入放入 P4。
- 异常注入先作用于虚拟执行体和消息总成，不对真实机器人连接开放。
- `/simulation` 页面只展示和调度运行，不编辑正式地图配置。

## 仿真驾驶舱页面优化方案

### 页面定位

页面路由建议保持：

```text
/simulation
```

页面定位：

- 单独页面，不嵌入地图编辑页面。
- 面向仿真运行、任务联调、消息诊断和异常演练。
- 使用 Apple + Linear 风格，保持高级感、极简、低噪声、高级 typography。
- 技术栈沿用 React、Tailwind、shadcn/ui、Framer Motion。
- 前端只通过 REST API 和 WebSocket 与平台交互，不直连 MQTT，不直接操作虚拟执行体。

### 页面核心流程

```text
选场景
-> 创建 SimulationRun
-> 建任务
-> 生成或选择 Plan
-> 创建 Action
-> 消息总成下发 MQTT command
-> 虚拟执行体返回 result
-> Message Hub 写入 MessageRecord
-> result event 标准化为 Observation
-> CurrentState 更新
-> 驾驶舱刷新状态、消息、Trace 和指标
-> 必要时注入异常或重规划
```

### 信息架构

建议采用“三栏 + 底部诊断区”的工作台布局：

```text
顶部运行栏
├─ Workspace / Run / 场景 / 运行状态 / 用时 / 操作按钮

左侧：场景与任务
├─ 选场景
├─ 建任务
├─ Task 模板
└─ Plan 步骤

中间：二维 CurrentState
├─ 只读二维场地
├─ 机器人位置
├─ 任务路径
├─ 工位 / 资源 / 阻塞点
└─ 当前异常标记

右侧：指令与状态
├─ Action 下发
├─ 机器人状态
├─ 任务进度
├─ 资源占用
└─ 异常注入

底部：诊断区 Tabs
├─ Messages
├─ Trace
├─ Observations
├─ Snapshots
├─ Metrics
└─ Export
```

### 1. 选场景

目标：

- 选择已经配置好的二维场地、机器人、动作集、任务流。
- 创建 `SimulationRun` 前必须绑定场景快照，保证本次运行可复现。

推荐场景对象：

```text
Scenario
├─ scenarioId
├─ name
├─ siteMapId
├─ siteMapVersion
├─ robotCodes[]
├─ robotTypeIds[]
├─ actionSetId
├─ taskFlowId
├─ resourceProfileId
└─ createdAt
```

页面表现：

- 左侧顶部使用场景选择器。
- 选择后展示场地图、机器人清单、动作集摘要、任务流摘要。
- 地图在驾驶舱中只读展示；如需编辑，跳转地图编辑页。

### 2. 建任务

目标：

- 用户可以手动创建模拟任务。
- 用户也可以从任务模板生成任务。

任务创建方式：

- 手动任务：填写目标点、起点、目标工位、货物类型、优先级、约束、超时时间。
- 模板任务：选择模板后填入必要参数，例如分拣、抓取、搬运、巡检、到点。

推荐状态：

```text
Draft -> Ready -> Running -> Succeeded
                    |-> Failed
                    |-> Cancelled
```

页面表现：

- 左侧使用紧凑表单创建 Task。
- 创建后展示 Task 摘要、Plan 版本和步骤列表。
- Task 与 Plan 分开展示，避免把业务目标和执行步骤混在一起。

### 3. 发指令

目标：

- 所有动作指令必须通过消息总成下发。
- 前端不直接操作执行体，不直连 MQTT，不直接改机器人状态。

推荐链路：

```text
驾驶舱页面
-> POST /api/v1/actions
-> Action Service 校验
-> Message Hub 创建 Command MessageRecord
-> MQTT Bridge 发布 factory/dogs/{robotCode}/command
-> 虚拟机器人执行体订阅 command
-> 虚拟机器人发布 result
-> Message Hub 接收 result
```

页面表现：

- 右侧提供 Action 下发面板。
- Action 下发面板只允许选择平台支持的动作，例如 `goto_pose`、`where`、`stop`。
- 每条 Action 必须显示 `actionId`、`commandId`、`traceId`、`status` 和耗时。

### 4. 看状态

目标：

- 查看机器人位置、状态、任务进度、资源占用。
- 状态来自 `CurrentState`，不是前端临时拼接。

状态内容：

- 机器人：位置、方向、电量、在线状态、当前动作、目标点、速度、最后心跳时间。
- 任务：Task 状态、Plan 版本、当前步骤、完成百分比、失败原因。
- 资源：工位占用、路径占用、资源锁、等待队列。
- 环境：障碍物、路径阻塞、异常区域、接口状态。

页面表现：

- 中间二维地图展示当前状态。
- 右侧使用状态摘要和资源占用列表。
- 状态变化使用低强度 Framer Motion 动画，避免过度装饰。

### 5. 看消息

目标：

- 查看 Command、Telemetry、Event、Ack、Alert 等消息流。
- 能按 Run、Task、Action、Robot、Trace、Topic、时间范围过滤。

推荐消息分类：

| 类型 | 含义 | 来源 |
|---|---|---|
| Command | 平台下发的动作指令 | Platform API / MQTT Bridge |
| Ack | 指令接收、拒绝、开始执行等确认 | MQTT result event |
| Telemetry | 位置、心跳、传感状态等运行遥测 | MQTT result event |
| Event | 任务开始、任务完成、动作完成等业务事件 | MQTT result event / 平台内部 |
| Alert | 离线、失败、超时、阻塞、消息丢失等异常 | MQTT result event / 异常注入 |

推荐 MQTT event 映射：

| MQTT result event | 标准分类 |
|---|---|
| `command.accepted` | Ack |
| `command.rejected` | Ack |
| `task.started` | Event |
| `task.succeeded` | Event |
| `task.failed` | Alert |
| `pose.updated` | Telemetry |
| `where.result` | Telemetry |
| `robot.offline` | Alert |
| `path.blocked` | Alert |
| `interface.timeout` | Alert |
| `message.dropped` | Alert |

页面表现：

- 底部 `Messages` Tab 作为主消息流。
- 支持快速过滤：`Command`、`Ack`、`Telemetry`、`Event`、`Alert`。
- 每条消息可展开查看 payload 摘要、topic、QoS、traceId、commandId、requestId。

### 6. 注入异常

目标：

- 模拟机器人离线、动作失败、路径阻塞、接口超时、消息丢失等异常。
- 异常必须进入统一事件链路，不能直接修改最终状态。

推荐异常类型：

```text
robot.offline
action.failed
path.blocked
interface.timeout
message.dropped
resource.locked
station.unavailable
battery.low
```

推荐链路：

```text
驾驶舱异常注入
-> POST /api/v1/simulation-runs/{runId}/events
-> 创建 Event / Observation
-> 必要时通知虚拟执行体进入异常模式
-> CurrentState 聚合异常影响
-> Agent 或规则策略判断重试、停止、等待或重规划
-> Trace 记录异常影响链路
```

页面表现：

- 右侧提供“异常注入”面板。
- 异常按对象选择：机器人、路径、工位、接口、消息。
- 每次注入必须显示影响范围、持续时间、是否自动恢复、traceId。
- 高风险异常需要二次确认，例如持续离线、全局消息丢失。

## 通信规范调整

P3 不改变机器狗 MQTT 基础契约：

```text
command topic: factory/dogs/{robotCode}/command
result topic:  factory/dogs/{robotCode}/result
```

P3 新增的是平台内部标准化层：

```text
MQTT command/result
-> MessageRecord
-> Observation
-> CurrentState
-> Trace / Snapshot / Export
```

所有驾驶舱下发动作必须满足：

- 必须创建 `Action`。
- 必须生成或绑定 `commandId`、`traceId`、`taskId`、`runId`。
- 必须通过平台 API 和 MQTT Bridge 发布。
- 不允许前端直连 MQTT Broker 发布 command。
- 不允许前端直接调用虚拟执行体。

## UI 设计要求

- 整体采用 Apple + Linear 风格。
- 页面首屏就是工作台，不做营销式 Hero。
- 使用 shadcn/ui 组件体系，按钮、Tabs、Badge、Tooltip、Dialog、Sheet、Table、Command Palette 保持统一。
- 使用 Tailwind 做响应式布局。
- 使用 Framer Motion 做状态切换、列表新增、面板展开和 Trace 定位动画。
- 地图为主要视觉中心，但不是编辑器。
- 状态和异常不能只依赖颜色，必须有图标和文本标签。
- 控制按钮使用图标优先，危险操作使用明确文案和确认。
- 桌面端优先三栏工作台，平板端折叠为双栏，手机端只保留观察、消息和关键操作。

## 回滚方案

- P3 所有数据库变更使用新的 Alembic migration，不修改 P1 已完成 migration。
- `/simulation` 页面作为新路由接入，出现问题时可从导航中隐藏，不影响现有地图编辑和 MQTT 联调页面。
- 新增 API 使用独立路由，不替换现有 `/api/v1/commands`、`/api/v1/messages`、`/api/v1/commands/{commandId}/trace`。
- Action 层失败时仍可保留现有命令调试入口作为联调兜底。
- Observation、CurrentState、Snapshot、Trace 出现聚合问题时，原始 `MessageRecord` 仍作为事实源保留，可用于重建。
