## 1. 规划约束

后续所有开发必须以本文档为执行基准。

执行原则：

- 不跳阶段：除非用户明确要求，否则按 P0 到 P6 顺序推进；已完成的 P2 能力作为现有基线保留。
- 不隐式扩展：新增需求必须先判断属于哪个阶段，再进入实施。
- 不破坏边界：前端不直接下发机器人控制指令；虚拟机器人执行体不依赖平台内部代码；智能体负责决策与指令规划。
- 不先编码后补文档：涉及架构、通信、数据库、部署、真实机器人接入的变更，必须先更新规划或规范文档。
- 不绕过验证：每个阶段完成后必须有构建、接口、通信或业务闭环验证结果。
- 不把临时实现当正式方案：MVP 中的 JSON 文件持久化只是过渡方案，正式阶段必须迁移到 PostgreSQL。

变更控制：

- 若新需求与本文档冲突，必须先修改本文档并说明原因。
- 若新需求跨越多个阶段，必须拆分为多个实施任务。
- 若发现当前阶段依赖未完成，不得直接进入下一阶段核心功能。

## 3. 阶段总览

| 阶段 | 名称 | 核心目标 | 优先级 |
|---|---|---|---|
| P0 | MVP 稳定化 | 让现有闭环稳定、可演示、可验证 | 最高 |
| P1 | 数据持久化标准化 | 从 JSON 文件迁移到 PostgreSQL 与 Redis | 高 |
| P2 | 消息总成与 MQTT 联调增强 | 形成标准联调、追踪、导出能力 | 高 |

## 4. P0：MVP 稳定化

### 4.1 目标

将当前 MVP 从“可运行原型”提升到“稳定演示版本”。

### 4.2 开发内容

- 前端接入 WebSocket 实时推送。
- 机器人状态在地图上实时刷新。
- 增加指令调试面板，仅用于测试 MQTT 链路。
- 增强地图草稿校验：
  - 对象越界。
  - 路径断链。
  - 重复 ID。
  - 路径边引用错误。
  - 基础碰撞检测。
- 增加地图导入格式校验与错误提示。
- 增加后端健康检查细分：
  - API 状态。
  - MQTT Bridge 状态。
  - 数据存储状态。
  - 虚拟执行体最近心跳。
- 增加基础单元测试和 API 测试。
- 增加前端基础交互验证。

### 4.3 不包含内容

- 不迁移数据库。
- 不实现正式智能体。
- 不实现复杂多机器人调度。
- 不接真实机器人。

### 4.4 验收标准

- 前端地图可实时显示机器人状态变化。
- WebSocket 断线后可恢复或回退到轮询。
- 地图非法配置可被后端拒绝并返回明确问题。
- 导入非法地图文件有可读错误提示。
- 指令调试面板可以触发一条 MQTT command 并看到 result event。
- 自动化测试覆盖核心 API。
- `docker compose up -d` 后核心服务可用。

## 5. P1：数据持久化标准化

### 5.1 目标

将 MVP 的 JSON 文件持久化迁移为正式数据库结构，建立可持续演进的数据底座。

### 5.2 开发内容

- 接入 PostgreSQL。
- 引入数据库 migration。
- 建立核心表：
  - 地图配置。
  - 地图草稿。
  - 地图编辑操作。
  - 地图校验结果。
  - 机器人实例。
  - 指令记录。
  - 动作运行记录。
  - MQTT 消息记录。
  - 事件记录。
  - 导入任务。
  - 导出任务。
  - 审计日志。
- 接入 Redis：
  - 最新机器人状态缓存。
  - 消息推送缓冲。
  - WebSocket 广播辅助。
- 保留 JSON 配置导入导出能力。
- 增加数据初始化脚本。
- 增加数据库备份与恢复说明。

### 5.3 不包含内容

- 不做复杂历史回放。
- 不做高频状态长期全量存储优化。
- 不引入 Kafka。

### 5.4 验收标准

- JSON 文件不再作为主存储。
- 地图配置、草稿、消息、事件、指令可落库。
- 配置版本可追踪。
- commandId、correlationId 可串联完整链路。
- Docker Compose 中 PostgreSQL 与 API 正常联通。
- migration 可重复执行并可在空库初始化。

## 6. P2：消息总成与 MQTT 联调增强

### 6.1 目标

将平台从“能收发 MQTT”提升到“能标准化联调、追踪和诊断”。

### 6.2 开发内容

- 完善 MQTT 连接配置。
- 增加 MQTT Contract 查询页面。
- 增加消息筛选：
  - robotId。
  - commandId。
  - correlationId。
  - messageType。
  - topic。
  - 时间范围。
- 增加链路追踪视图：
  - command。
  - event。
  - pose.updated。
  - task.started。
  - task.succeeded/task.failed。
- 增加 retained 消息检查。
- 增加 Last Will 离线事件验证。
- 增加 MQTT 联调导出包。
- 增加 Topic 权限模型设计。
- 增加 MQTT 鉴权与 TLS 部署规划。

### 6.3 不包含内容

- 不直接控制真实机器人。
- 不让前端直接连接 command Topic。
- 不把消息总成变成调度系统。

### 6.4 验收标准

- 任意 command 可追踪到 result event。
- MQTT 消息可按关键字段筛选。
- 能导出单次联调完整消息包。
- Last Will 可产生离线事件并被平台记录。
- MQTT Contract 页面显示当前 Topic、QoS、Retain、Payload 约束。

## 7. P3：统一运行模型与仿真驾驶舱

### 7.1 目标

建立 SimulationRun、Task、Plan、Action、Observation、CurrentState、Snapshot、Trace，并提供独立 `/simulation` 驾驶舱。

### 7.2 开发内容

- 建立 Run、Task、Plan、Plan Step、Action 状态机。
- 将 MQTT result event 标准化为 Observation。
- 建立可重建、带 `stateVersion` 的 CurrentState。
- 建立关键检查点 Snapshot。
- 建立 Trace Header 和 Span。
- 建设驾驶舱：
  - Run 和 Task 控制。
  - Plan 版本和步骤。
  - 二维 CurrentState。
  - Action 和 Observation。
  - Trace、Messages、Snapshot、Metrics、Decision Log、Export。
- 支持观察、手动联调、任务流、智能体和回放模式。

### 7.3 不包含内容

- 不训练 AI 模型。
- 不实现三维物理仿真。
- 不允许前端直连 MQTT 或直接修改 CurrentState。

### 7.4 验收标准

- Task 可追踪到 Plan、Action、command、result、Observation 和最终状态。
- Plan 重规划产生新版本。
- Snapshot 引用明确 `stateVersion` 并可用于只读回放。
- Trace 可解释任务成功、失败、等待和重规划。
- 单次 Run 可导出完整且不包含 Secret 的联调包。

## 8. P4：智能体与复杂仿真

### 8.1 目标

在统一运行模型上实现智能体闭环、多机器人、资源占用、随机事件和恢复策略。

### 8.2 开发内容

- 新增独立 `agent-service`。
- 实现规则 Agent 和 AI Agent 适配接口。
- Agent 只读取 Observation、CurrentState、Task 和 Plan，不直接修改机器人状态。
- AI 输出必须经过 Action 校验后进入 command API。
- 支持多机器人执行体、机器人类型、动作耗时区间和资源锁。
- 支持路径阻塞、工位不可用、设备异常、网络抖动和耗时波动。
- 支持等待、失败、超时、重试、停止和重规划。
- 记录 Observation、Decision、Validation、Latency、Action 和 Plan Version。

### 8.3 不包含内容

- 不训练 AI 模型。
- 不实现真实导航和避障算法。
- 不将消息总成变成调度系统。

### 8.4 验收标准

- Agent 可根据 CurrentState 生成或更新 Plan。
- Action 经过平台校验并可追踪到 MQTT result。
- 多机器人资源冲突不会产生非法重复占用。
- 随机事件先生成 Observation，再影响 CurrentState 和 Plan。
- Agent 不可用时可回退规则策略。

## 9. P5：多用户与 1～20 Workspace

### 9.1 目标

建立共享控制平面，每个用户默认拥有独立私有 Workspace，每 Workspace 使用独立运行栈。

### 9.2 开发内容

- Tenant、User、Workspace、WorkspaceMember 和 WorkspaceInstance。
- Keycloak/OIDC/JWT 和 owner、operator、viewer、auditor 角色。
- Control Plane：创建、启动、停止、升级、归档和删除 Workspace。
- 每 Workspace 独立 Frontend、Platform API、Agent、MQTT Broker、Executor、Redis、Network 和 Volume。
- 共享 PostgreSQL Server，每 Workspace 独立 Database 和 Role。
- Traefik/Nginx HTTP 路由和 MQTT 端口池 `18830～18849`。
- Workspace 配额、健康、日志、备份、审计和镜像升级。

### 9.3 不包含内容

- 不复制代码仓库。
- 不引入 Kubernetes、Service Mesh 或跨宿主机高可用。
- 不允许 Workspace 直接访问其他 Workspace 网络或数据。

### 9.4 验收标准

- 可同时运行至少 5 个 Workspace，设计容量达到 20 个。
- 两个 Workspace 使用相同 `robotCode` 时互不影响。
- 用户无法访问未授权 Workspace 的页面、API、MQTT、Snapshot、Trace 和文件。
- Workspace 停止后数据可恢复，统一镜像可批量升级。

## 10. P6：真实机器人接入准备

### 10.1 目标

让真实机器人网关替换指定 Workspace 的虚拟执行体，平台主体不需要重构。

### 10.2 开发内容

- 真机 MQTT 接入和 Workspace 绑定说明。
- MQTT 用户名/证书、ACL、TLS、指令白名单和急停策略。
- 真实机器人同一时间只绑定一个 Workspace。
- 虚拟执行体与真实机器人切换和凭证轮换。
- 真实机器人联调用例、记录和导出包。

### 10.3 不包含内容

- 不实现具体厂商底盘控制、SLAM、导航或避障算法。

### 10.4 验收标准

- 真实机器人实现 command/result 契约后可接入指定 Workspace。
- command 可到达真实网关，result 可更新 Observation 和 CurrentState。
- 解绑后旧凭证失效，其他 Workspace 不受影响。

## 13. 推荐下一轮任务

P1 已完成验收，下一轮进入 P3，优先任务如下：

1. 建立 SimulationRun、Task、Plan、Plan Step、Action 数据结构和 Migration。
2. 建设仿真驾驶舱 C0 只读工作台。
3. 将 MQTT result event 标准化为 Observation。
4. 建立 CurrentState `stateVersion` 和 Trace 基础结构。
5. 建立 Snapshot 检查点和单次 Run 导出结构。

推荐原因：

- 驾驶舱依赖稳定的运行数据模型，不能继续基于 JSON 文件扩展。
- Observation、CurrentState、Snapshot 和 Trace 是后续 Agent、多机器人和 Workspace 的共同基础。
- 先完成单 Workspace 运行模型，再进入 P5 多用户编排，可降低隔离改造风险。

## 19. v2.0 多 Workspace 与仿真驾驶舱文档治理：2026-06-22

Trace 贯穿全部对象
