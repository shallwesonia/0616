# 仿真驾驶舱优化建议整合方案

## 1. 页面定位

仿真驾驶舱是面向研发验证、方案演示、接口联调、异常演练和后续智能体调度测试的综合模拟页面，不是普通监控大屏，也不是地图编辑器。

页面核心边界：

- 前端只负责配置选择、交互发起、过程展示和结果查看。
- 前端不直接修改机器人运行状态。
- 前端不直连 MQTT command Topic。
- 所有动作指令必须通过消息总成下发。
- 所有状态变化必须来自虚拟机器人执行体、仿真引擎、消息总成或后端运行事实源。
- 所有 Command、Telemetry、Event、Ack、Alert、Interface、AgentDecision 必须留痕。
- 后续接入真实机器人或机器人网关时，应复用同一套消息协议、页面逻辑和追踪链路。

页面路由保持：

```text
/simulation
```

## 2. 核心闭环

仿真驾驶舱围绕六个核心目标形成完整闭环：

```text
选场景
-> 建任务
-> 发指令
-> 看状态
-> 看消息
-> 注入异常
-> 观察任务流、资源、机器人、消息和智能体反应
-> 生成仿真结果
-> 优化场景、任务流、动作耗时、资源规则和调度策略
```

闭环验证目标：

- 验证二维场景配置是否合理。
- 验证任务流是否可跑通。
- 验证机器人动作集是否完整。
- 验证动作耗时是否符合预期。
- 验证资源占用是否有冲突。
- 验证消息总成和 MQTT 链路是否完整。
- 验证异常处理是否可靠。
- 验证智能体是否能正确接入。
- 验证后续真实机器人替换虚拟执行体的可行性。

## 3. 页面信息架构

建议采用“顶部运行栏 + 三栏工作台 + 底部诊断区”。

```text
顶部运行栏
├─ Workspace
├─ SimulationRun
├─ 当前场景
├─ 运行状态
├─ 运行时长
├─ 时间倍率
├─ 随机事件策略
└─ 启动 / 暂停 / 恢复 / 停止 / 导出

左侧：场景与任务
├─ 场景选择
├─ 场景校验
├─ 任务模板
├─ 手动建任务
├─ 批量建任务
└─ Task / Plan / Step

中间：二维 CurrentState
├─ 只读二维地图
├─ 坐标系
├─ 机器人位置
├─ 路径与轨迹
├─ 工位和资源状态
├─ 阻塞点
└─ 当前异常标记

右侧：指令、状态与异常
├─ 快捷指令
├─ 高级指令
├─ 机器人状态
├─ 任务进度
├─ 资源占用
├─ 异常注入
└─ 恢复操作

底部诊断区
├─ Messages
├─ Trace
├─ Observations
├─ Snapshots
├─ Metrics
├─ Decision Log
└─ Export
```

## 4. 选场景

### 4.1 功能定位

选场景是仿真运行起点。场景不是单一二维地图，而是一组可运行配置集合。

场景应包含：

| 配置项 | 内容 |
|---|---|
| 二维场地 | 场地尺寸、坐标系、区域、障碍物、工位、路径、点位 |
| 区域配置 | 工作区、待机区、缓存区、充电区、禁行区、操作区 |
| 工位配置 | 装载、卸载、分拣、检测、缓存等工位 |
| 路径配置 | 路径节点、路径边、方向、容量、通行规则 |
| 机器人类型 | 机器狗、AMR、搬运机器人、分拣机器人等 |
| 机器人实例 | robotCode、初始位置、初始状态、绑定类型 |
| 动作集 | 移动、抓取、放置、装载、卸载、检测、等待、充电 |
| 动作耗时 | 固定耗时、区间耗时、距离耗时、随机扰动耗时 |
| 任务模板 | 转运、装箱、卸货、巡检、分拣等业务流程 |
| 资源规则 | 工位占用、路径占用、缓存容量、充电桩占用 |
| 消息规则 | Command、Telemetry、Event、Ack、Alert |
| 调度策略 | 手动调度、规则调度、智能体调度 |
| 异常规则 | 故障、超时、消息丢失、接口失败、资源不可用 |

### 4.2 场景校验

创建 Run 前必须校验：

- 地图是否存在。
- 任务引用的工位是否存在。
- 起点到终点是否有可用路径。
- 场景是否有可执行机器人实例。
- 机器人类型是否支持任务动作。
- 资源规则是否完整。
- 任务模板是否有效。
- 消息协议是否匹配。
- 调度策略是否可用。
- 初始位置、资源占用和机器人状态是否冲突。

### 4.3 交互要求

- 支持选择已有场景。
- 展示场景绑定的地图、机器人、任务模板和动作集。
- 展示地图版本、机器人配置版本、动作集版本、任务模板版本。
- 创建 `SimulationRun` 时记录场景快照，保证可复现。
- 配置缺失时明确提示缺失对象。

## 5. 建任务

### 5.1 功能定位

建任务用于在选定场景下创建一条或多条模拟任务。任务是机器人、资源、消息和状态变化的核心驱动对象。

### 5.2 创建方式

| 方式 | 说明 | 使用场景 |
|---|---|---|
| 模板创建 | 从已有任务模板生成任务 | 标准流程验证 |
| 手动创建 | 手动选择机器人、起点、终点、动作和约束 | 单动作或单流程调试 |
| 批量创建 | 按数量、频率、优先级批量生成任务 | 压力测试、吞吐验证 |

### 5.3 手动任务字段

手动任务建议支持：

- 任务名称。
- 任务类型。
- 执行机器人，可指定或自动分配。
- 起点。
- 终点。
- 动作序列。
- 资源需求。
- 优先级。
- 超时时间。
- 重试策略。
- 完成条件。

### 5.4 批量任务字段

批量任务建议支持：

- 任务数量。
- 任务生成间隔。
- 优先级分布。
- 起点和终点范围。
- 是否随机选择机器人。
- 是否随机任务类型。
- 是否启用随机扰动。
- 是否启用异常概率。
- 随机种子。
- 是否自动运行。

### 5.5 任务状态机

建议统一为：

```text
Created
-> Queued
-> Assigned
-> Running
-> Waiting
-> Succeeded
```

异常分支：

```text
Running -> Failed
Running -> Timeout
Running -> Cancelled
Waiting -> Running
Failed -> Queued
```

### 5.6 任务事件

任务创建和运行至少产生：

- `TASK_CREATED`
- `TASK_QUEUED`
- `TASK_ASSIGNED`
- `TASK_STARTED`
- `TASK_STEP_STARTED`
- `TASK_STEP_COMPLETED`
- `TASK_COMPLETED`
- `TASK_FAILED`

## 6. 发指令

### 6.1 功能定位

发指令用于人工触发动作指令，也用于查看智能体或调度器下发的指令。

强制边界：

- 页面不能直接调用机器人执行体。
- 页面不能直接修改机器人状态。
- 页面不能直接发布 MQTT command。
- 所有指令先进入 `Action`。
- `Action` 经消息总成校验、记录、分发后，再由 MQTT Bridge 发布。

### 6.2 指令类型

建议支持：

- 任务分配指令。
- 移动指令。
- 动作执行指令。
- 暂停指令。
- 恢复指令。
- 取消指令。
- 重试指令。
- 资源释放指令。
- 异常处理指令。
- 查询指令。

### 6.3 下发入口

建议分两类：

快捷指令：

- 移动到装载工位。
- 移动到卸载工位。
- 执行装载。
- 执行卸载。
- 暂停当前任务。
- 恢复当前任务。
- 重试当前动作。
- 取消当前任务。

高级指令：

- 指令类型。
- robotCode。
- taskId。
- actionCode。
- 目标工位。
- 目标路径。
- 执行参数。
- timeoutMs。
- retryCount。
- priority。

### 6.4 指令校验

下发前必须校验：

- 指令格式完整。
- robotCode 存在。
- 机器人状态允许执行。
- 目标点合法。
- 路径可达。
- 资源可用。
- 动作集支持该动作。
- 超时时间合法。
- 是否重复 command。
- 用户是否有操作权限。

### 6.5 标准链路

```text
页面创建 Action
-> Action Service 校验
-> Message Hub 记录 Command
-> MQTT Bridge 发布 command
-> 虚拟机器人执行体接收
-> 执行体返回 Ack / Telemetry / Event / Alert
-> Message Hub 记录 result
-> Observation
-> CurrentState
-> Trace
```

## 7. 看状态

### 7.1 状态来源

页面状态必须来自 `CurrentState`，不能由前端临时拼接作为事实源。

### 7.2 状态范围

需要展示：

- Run 状态：运行中、暂停、完成、失败、停止。
- Task 状态：排队、已分配、运行、等待、完成、失败、超时。
- Plan 状态：版本、当前步骤、重规划原因。
- Action 状态：Pending、Issued、Accepted、Running、Succeeded、Failed、Timeout。
- Robot 状态：位置、方向、电量、在线状态、当前动作、目标点、最后心跳。
- Resource 状态：工位占用、路径占用、资源锁、等待队列。
- Environment 状态：阻塞点、禁行区、异常区域。
- Interface 状态：上游、下游、MQTT、WebSocket、Agent。

### 7.3 地图展示

二维地图中应展示：

- 坐标轴。
- 只读场景地图。
- 机器人当前位置。
- 目标点。
- 规划路径。
- 已走轨迹。
- 工位状态。
- 路径容量和阻塞。
- 资源锁。
- 异常标记。

## 8. 看消息

### 8.1 消息类型

消息总成展示应覆盖：

| 类型 | 说明 |
|---|---|
| Command | 平台下发的动作指令 |
| Telemetry | 机器人位置、状态、电量、心跳 |
| Event | 任务开始、动作完成、资源释放 |
| Ack | 指令已接收、已拒绝、已开始执行 |
| Alert | 动作失败、超时、机器人离线 |
| Interface | 上下游任务下发、外部系统回调 |
| AgentDecision | 智能体任务分配、动作选择、重规划 |

### 8.2 消息字段

消息列表建议展示：

- 时间。
- messageId。
- traceId。
- 类型。
- 来源。
- 目标。
- topic。
- taskId。
- robotCode。
- 状态。
- payload 摘要。
- 操作：查看详情、复制、重放、导出。

### 8.3 消息详情

消息详情应包含：

- 基本信息。
- payload。
- 关联 Task。
- 关联 Robot。
- 上一条关联消息。
- 下一条关联消息。
- 处理状态。
- 发送时间。
- 接收时间。
- 处理耗时。
- 错误原因。
- 重试记录。

### 8.4 Trace 链路

消息必须支持按 `traceId` 串联。

典型链路：

```text
CMD_MOVE
-> ACK_ACCEPTED
-> STATUS_MOVING
-> POSITION_UPDATED
-> ACTION_COMPLETED
-> STATUS_IDLE
```

页面需要回答：

- 指令是否发出。
- 机器人是否收到。
- 机器人是否开始执行。
- 执行过程中是否上报状态。
- 动作是否完成。
- 任务是否进入下一步。
- 失败发生在哪个环节。

### 8.5 消息过滤、重放与导出

过滤条件：

- 时间范围。
- 消息类型。
- robotCode。
- taskId。
- traceId。
- topic。
- 状态。
- 来源。
- event。

消息重放：

- 支持重放单条消息。
- 支持重放某个 Task 的完整消息链。
- 支持重放某个时间段消息。
- 重放必须进入沙箱模式。
- 重放不得影响真实系统。

导出：

- 当前过滤结果。
- 某个 Task 的完整消息。
- 某个 Robot 的完整消息。
- 某次 Run 的全部消息。
- 异常相关消息。

## 9. 注入异常

### 9.1 功能定位

异常注入用于人为制造异常，验证平台、任务流、机器人状态机、资源占用、消息总成和智能体调度的异常处理能力。

### 9.2 异常类型

| 类型 | 示例 | 验证目标 |
|---|---|---|
| 机器人异常 | 离线、故障、电量不足、状态卡死 | 机器人异常处理 |
| 动作异常 | 抓取失败、放置失败、装载失败、移动失败 | 动作重试和失败流程 |
| 路径异常 | 路径阻塞、容量不足、禁行 | 路径资源和重调度 |
| 工位异常 | 工位占用、工位不可用、工位故障 | 等待、排队、任务切换 |
| 消息异常 | 延迟、丢失、重复、乱序 | 消息总成稳定性 |
| 接口异常 | 上游超时、下游失败、回调异常 | 接口适配和重试 |
| 智能体异常 | 非法指令、无效决策、重复调度 | 指令校验和规则兜底 |
| 资源异常 | 缓存位满、充电桩占用、料箱缺失 | 资源占用和任务等待 |

### 9.3 注入方式

立即注入：

- 立即让某机器人离线。
- 立即让某路径阻塞。
- 立即让某工位不可用。
- 立即让当前动作失败。
- 立即丢弃下一条状态消息。

条件注入：

- 机器人到达某工位时注入工位不可用。
- 任务执行到第 N 步时注入动作失败。
- command 发出后延迟 Ack。
- 机器人进入某路径时触发路径阻塞。
- 任务运行超过阈值后触发接口超时。

### 9.4 注入配置

每个异常支持：

- 异常名称。
- 异常类型。
- 目标对象。
- 触发方式。
- 触发时间。
- 持续时间。
- 恢复方式。
- 影响范围。
- 严重等级。
- 是否写入事件和审计日志。

### 9.5 恢复机制

恢复方式包括：

- 自动恢复。
- 手动恢复。
- 重试恢复。
- 重新调度。
- 跳过步骤。
- 人工接管。
- 终止任务。

### 9.6 异常事件

异常至少产生：

- `FAULT_INJECTED`
- `ROBOT_OFFLINE`
- `ACTION_FAILED`
- `PATH_BLOCKED`
- `STATION_UNAVAILABLE`
- `MESSAGE_DROPPED`
- `MESSAGE_DELAYED`
- `INTERFACE_TIMEOUT`
- `FAULT_RECOVERED`
- `TASK_REPLANNED`
- `TASK_FAILED`

## 12. 最终目标

仿真驾驶舱最终应成为二维具身智能仿真平台的综合模拟入口。

它不只是“看机器人动起来”，而是要完成：

- 通过选场景确定仿真边界。
- 通过建任务生成业务流程。
- 通过发指令验证消息下发链路。
- 通过看状态验证机器人和资源变化。
- 通过看消息验证接口、事件和回执闭环。
- 通过注入异常验证系统韧性。
- 通过日志和指标支撑流程优化。
- 通过统一接口支撑后续 AI 智能体调度接入。
- 通过虚拟机器人执行体支撑后续真实机器人网关平滑替换。

## 13. 语义澄清补充

### 13.1 停止动作

`POST /api/v1/actions/{actionId}/stop` 与 `command=stop` 必须区分：

- `/actions/{actionId}/stop`：停止平台侧某条 Action 的生命周期，不默认发布 MQTT。
- `command=stop`：新建 stop Action，经消息总成向机器人下发停止指令。

若要完成“机器人停止 + 原 Action 结束”，应由 stop Action 的 result 事件驱动原 Action 收敛，而不是前端直接改状态。

### 13.2 Sandbox 回放

当前 sandbox replay 不发布 MQTT，不驱动执行体，但会写入当前 Workspace 的调试记录。该能力在隔离回放环境完成前不应作为普通驾驶舱操作暴露给用户。

### 13.3 robot.offline

`robot.offline` 当前只代表仿真控制台注入的故障发生。正式环境中，机器人离线事实必须来自机器人端、网关或心跳监控上报，平台统一归类为 `Alert`。
