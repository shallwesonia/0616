## 1. 标准目标

本文档用于统一平台在本机、Docker 网络和局域网内的连接方式、端口、协议、Topic、访问边界和联调约束。

核心原则：

- 前端只访问 HTTP REST 和 WebSocket，不直接连接机器人 MQTT command Topic。
- 机器人执行体只依赖 MQTT 契约，不依赖平台内部代码。
- 智能体或调试面板下发动作必须通过平台 API 进入消息总成。
- MQTT command 消息禁止 retained。
- 真实机器人网关后续必须复用 `factory/dogs/{robotCode}/command` 与 `factory/dogs/{robotCode}/result` 契约。
- HTTP 通过共享反向代理路由到当前 Workspace，业务 API 必须校验 JWT `workspaceId`。
- 每 Workspace 使用独立 MQTT Broker，避免修改机器狗终版 Topic 和 Payload。
- Workspace 内部服务不直接暴露到局域网，只有反向代理和分配的 MQTT 端口对外开放。

## 2. 1～20 Workspace 局域网暴露

共享入口：

| 服务 | 协议 | 默认地址 | 说明 |
|---|---|---|---|
| Reverse Proxy | HTTP/HTTPS | `http(s)://{PUBLIC_HOST}` | 所有 Workspace HTTP 入口 |
| Control Plane | HTTP REST | `http(s)://{PUBLIC_HOST}/control` | 登录、Workspace 和实例管理 |
| Identity | OIDC | `http(s)://{PUBLIC_HOST}/auth` | Keycloak/OIDC |
| Workspace MQTT | MQTT | `{PUBLIC_HOST}:18830～18849` | 每 Workspace 独立 Broker |

Workspace 推荐使用子域名：

```text
https://{workspaceSlug}.sim.local
https://api-{workspaceSlug}.sim.local
```

局域网没有 DNS 时使用路径路由：

```text
http://{PUBLIC_HOST}/w/{workspaceSlug}
http://{PUBLIC_HOST}/api/w/{workspaceSlug}
```

单 Workspace 本地开发继续兼容：

```text
http://{PUBLIC_HOST}:5173
http://{PUBLIC_HOST}:8000/api/v1
{PUBLIC_HOST}:18830
```

`PUBLIC_HOST` 应设置为宿主机局域网 IP，例如：

```text
192.168.1.100
```

## 3. 环境变量

控制平面基础配置：

```text
LAN_BIND_ADDRESS=0.0.0.0
PUBLIC_HOST=192.168.1.100
WORKSPACE_MQTT_PORT_START=18830
WORKSPACE_MQTT_PORT_END=18849
CONTROL_DATABASE_URL_FILE=/run/secrets/control_database_url
OIDC_ISSUER_URL=http://identity:8080/realms/simulation
```

每 Workspace 必须注入：

```text
TENANT_ID
OWNER_USER_ID
WORKSPACE_ID
WORKSPACE_SLUG
FRONTEND_PUBLIC_URL
API_PUBLIC_URL
MQTT_PUBLIC_HOST
MQTT_PUBLIC_PORT
MQTT_INTERNAL_HOST=mqtt-broker
MQTT_INTERNAL_PORT=1883
DATABASE_URL_FILE
REDIS_URL
WORKSPACE_DATA_PATH
IMAGE_VERSION
```

字段说明：

| 变量 | 默认值 | 说明 |
|---|---|---|
| `LAN_BIND_ADDRESS` | `0.0.0.0` | 服务监听地址。局域网暴露使用 `0.0.0.0` |
| `PUBLIC_HOST` | `localhost` | 对外展示的访问主机名或局域网 IP |
| `WORKSPACE_MQTT_PORT_START` | `18830` | Workspace MQTT 端口池起点 |
| `WORKSPACE_MQTT_PORT_END` | `18849` | Workspace MQTT 端口池终点 |
| `WORKSPACE_ID` | 无 | Workspace 隔离标识 |
| `MQTT_PUBLIC_PORT` | 动态分配 | 当前 Workspace MQTT 端口 |

## 4. 连接信息 API

平台提供连接信息接口：

```text
GET /api/v1/connections
```

该接口运行在 Workspace API 内，只返回当前 Workspace 的连接信息。控制平面查询指定 Workspace：

```text
GET /api/control/v1/workspaces/{workspaceId}/connections
```

用途：

- 前端或联调人员查询当前平台暴露地址。
- 真实机器人网关读取 MQTT Host、Port 和 Topic 模式。
- 联调脚本读取标准入口，避免硬编码端口。

返回结构包含：

- frontend URL。
- API Base URL。
- OpenAPI URL。
- WebSocket URL。
- MQTT Host、Port、Topic Prefix。
- command、result Topic。
- 支持的 command 清单。
- 标准 result event 清单。
- 通信规则。
- `workspaceId`、Workspace 状态和当前镜像版本。

连接信息不得返回 MQTT 密码、JWT、数据库连接串或证书私钥。

## 5. REST API 标准

Workspace Base URL：

```text
https://api-{workspaceSlug}.sim.local/api/v1
```

路径路由兼容地址：

```text
http://{PUBLIC_HOST}/api/w/{workspaceSlug}/api/v1
```

每个请求必须同时满足用户授权和 JWT `workspaceId` 匹配。

关键接口：

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/health` | 健康检查 |
| GET | `/connections` | 连接与协议信息 |
| GET | `/maps/current` | 当前地图 |
| POST | `/maps/{mapId}/drafts` | 保存地图草稿 |
| POST | `/maps/{mapId}/drafts/{draftId}/validate` | 校验草稿 |
| POST | `/maps/{mapId}/drafts/{draftId}/publish` | 发布草稿 |
| POST | `/commands` | 创建机器人动作指令 |
| GET | `/commands/{commandId}/trace` | 查询 command 链路 |
| GET | `/robots` | 查询机器人状态 |
| GET | `/messages` | 查询消息 |
| POST | `/events` | 触发控制台事件 |
| GET | `/mqtt/contract` | 查询 MQTT 契约 |

## 6. WebSocket 标准

前端优先连接：

```text
wss://api-{workspaceSlug}.sim.local/ws/v1/workspaces/{workspaceId}/runs/{runId}
```

后端直连地址：

```text
ws://platform-api:8000/ws/v1/workspaces/{workspaceId}/runs/{runId}
```

现有 `/ws/v1/sessions/{sessionId}` 作为 MVP 兼容路径，P3 驾驶舱完成后废弃。

当前消息类型：

```json
{
  "messageId": "ws-xxx",
  "type": "snapshot",
  "sessionId": "session-local",
  "timestamp": "2026-06-18T00:00:00Z",
  "data": {
    "robots": [],
    "messages": []
  }
}
```

规则：

- WebSocket 只用于状态展示。
- 前端不得通过 WebSocket 下发机器人动作指令。
- WebSocket 断开后前端必须回退到 REST 轮询。

## 7. MQTT 标准

Workspace 局域网地址：

```text
{PUBLIC_HOST}:{WORKSPACE_MQTT_PUBLIC_PORT}
```

容器内部地址：

```text
mqtt-broker:1883
```

每个 Workspace 拥有独立 Broker 容器和独立端口，端口由控制平面从 `18830～18849` 分配。

Topic Prefix：

```text
factory/dogs
```

标准 Topic：

| Topic | 方向 | QoS | Retain |
|---|---|---:|---|
| `factory/dogs/{robotCode}/command` | 平台 -> 机器狗 | 1 | false |
| `factory/dogs/{robotCode}/result` | 机器狗 -> 平台 | 1 | false |

### 7.1 MQTT 联调参数

| 参数 | 标准值 |
|---|---|
| Host | `{PUBLIC_HOST}` |
| Port | `{WORKSPACE_MQTT_PUBLIC_PORT}`，由控制平面从端口池分配 |
| Protocol | MQTT `3.1.1` |
| Keep Alive | `30s` |
| Clean Session | `true` |
| Username/Password | 本地联调默认关闭，生产环境必须开启 |
| TLS | 本地联调默认关闭，生产环境建议开启 |

Windows PowerShell 下发布 JSON Payload 时，推荐使用 stdin 模式，避免命令行引号转义破坏 JSON：

```powershell
$payload | docker exec -i 0616-mqtt-broker-1 mosquitto_pub `
  -h host.docker.internal `
  -p $env:WORKSPACE_MQTT_PUBLIC_PORT `
  -t factory/dogs/robot-001/result `
  -q 1 `
  -l
```

联调判定标准：

- MQTT 客户端可连接 `{PUBLIC_HOST}:{WORKSPACE_MQTT_PUBLIC_PORT}`。
- 平台 API `/api/v1/health` 中 `mqttBridge.connected=true`。
- 平台 API `/api/v1/messages` 可查询到外部 MQTT 客户端发布的合法 `event` result 消息。
- 非法 JSON Payload 会被记录为 `mqtt.payload.invalid` 系统消息。

## 8. MQTT command/result Payload

平台向机器狗下发 command：

```json
{
  "schemaVersion": "1.0",
  "messageType": "command",
  "commandId": "CMD-20260528-000001",
  "taskId": "TASK-20260528-000001",
  "requestId": null,
  "robotCode": "DOG-001",
  "traceId": "TRACE-20260528-000001",
  "command": "goto_pose",
  "issuedAt": "2026-05-28T10:00:00+08:00",
  "timeoutMs": 120000,
  "operatorId": "scheduler",
  "source": "scheduler",
  "params": {
    "x": 12.34,
    "y": 56.78
  }
}
```

机器狗向平台回传 result：

```json
{
  "schemaVersion": "1.0",
  "messageType": "event",
  "event": "task.succeeded",

  "requestId": null,

  "traceId": "TRACE-20260528-000001",
  "source": "device",
  "timestamp": "2026-06-18T00:00:00Z",

    "result": 1,
    "x": 12.34,
    "y": 56.78,
    "yaw": 1.57,
    "battery": 80
  },

```

## 9. Command 链路追踪

查询接口：

```text
GET /api/v1/commands/{commandId}/trace
GET /api/v1/traces/{traceId}
GET /api/v1/traces/{traceId}/spans
```

用于串联：

- command。
- result event。
- pose.updated。
- task.started。
- task.succeeded/task.failed。

追踪依据：

- `commandId`。
- `traceId`。
- `taskId`。
- `requestId`。

## 10. Workspace 暴露流程

1. 控制平面接收创建 Workspace 请求。
2. 校验 Workspace 名称、用户权限和资源配额。
3. 分配 `workspaceId`、`workspaceSlug`、Database、Role、Network 和 Volume。
4. 从 `18830～18849` 分配未占用 MQTT 端口。
5. 生成 Frontend/API 路由和 Workspace Secret。
6. 使用统一镜像启动 Workspace Docker Compose Stack。
7. 验证 API、Database、Redis、MQTT Broker、Executor 和 WebSocket。
8. 健康检查通过后将 Workspace 标记为 `Running`。
9. 用户通过控制平面获取当前 Workspace 访问入口。

控制平面接口：

```text
POST /api/control/v1/workspaces
POST /api/control/v1/workspaces/{workspaceId}/start
GET  /api/control/v1/workspaces/{workspaceId}/health
GET  /api/control/v1/workspaces/{workspaceId}/connections
```

单 Workspace 开发环境仍可执行：

```bash
docker compose up -d --build
```

但该方式不代表多用户正式部署标准。

## 11. 安全边界

- MQTT `allow_anonymous=true` 只允许单 Workspace 本地开发使用。
- 多 Workspace 环境必须启用 MQTT 用户名密码或证书和 ACL。
- API、WebSocket 和文件下载必须验证 OIDC/JWT、成员关系和 `workspaceId`。
- PostgreSQL、Redis、Workspace 内部 API 不得直接暴露到局域网。
- Workspace Docker Network 之间禁止互访。
- 生产环境启用 HTTPS、WSS 和 MQTT TLS。
- Secret 不得返回前端、写入日志、Snapshot 或导出包。
- 局域网暴露前必须配置 Windows/Linux 防火墙端口白名单。
