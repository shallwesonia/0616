# 数据库备份恢复与初始化标准

## 文档信息

| 项目 | 内容 |
|---|---|
| 文档类型 | PostgreSQL 运维标准 |
| 版本 | v1.0 |
| 日期 | 2026-06-22 |
| 适用范围 | 单 Workspace 基线及后续 1～20 Workspace |
| 权威数据源 | PostgreSQL |
| 辅助缓存 | Redis，可从 PostgreSQL 重建 |

## 1. 基本原则

- PostgreSQL 是唯一权威主存储；Redis 和 `data/state.json` 不参与数据库恢复决策。
- Redis 不备份运行态缓存，恢复 PostgreSQL 后由 Platform API 自动预热。
- 恢复必须先进入新建隔离数据库验证，禁止直接覆盖运行库。
- 迁移前、版本升级前、批量导入前必须创建逻辑备份。
- 备份文件不得包含在 Git 中，不得通过 API 或前端公开下载。
- 数据库密码只通过 `.env` 或密钥管理提供，不写入备份文件名、日志和文档。

## 2. 初始化流程

Docker 环境初始化顺序：

1. PostgreSQL 健康检查通过。
2. Platform API 执行 `alembic upgrade head`。
3. 若当前 Workspace 的 `config.site_maps` 为空，从 `data/state.json` 导入一次。
4. Platform API 从 PostgreSQL 预热 Redis 机器人状态和最近消息。
5. Uvicorn 开始提供 API 与 WebSocket 服务。

Migration 当前基线：`20260622_0001`。

初始化校验：

```bash
docker compose exec -T postgres psql -U sim -d sim_platform -c "SELECT version_num FROM alembic_version;"
docker compose exec -T postgres psql -U sim -d sim_platform -c "SELECT count(*) FROM config.site_maps WHERE status='active';"
```

## 3. 备份策略

| 类型 | 触发条件 | 建议保留 |
|---|---|---|
| 日常逻辑备份 | 每日低峰 | 最近 7 份 |
| 周备份 | 每周 | 最近 4 份 |
| 变更前备份 | Migration、升级、批量导入前 | 至少保留至变更验收完成 |
| Workspace 归档备份 | Workspace 归档或删除前 | 按项目归档周期 |

文件命名：`{database}_{yyyyMMdd_HHmmss}_{reason}.dump`。

当前 Docker 部署将宿主机 `data/backups` 挂载为 PostgreSQL 容器的 `/backups`。

执行自定义格式逻辑备份：

```bash
docker compose exec -T postgres pg_dump -U sim -d sim_platform -Fc -f /backups/sim_platform_20260622_prechange.dump
docker compose exec -T postgres pg_restore -l /backups/sim_platform_20260622_prechange.dump
```

备份成功标准：

- 命令退出码为 0。
- 宿主机 `data/backups` 中存在非空 `.dump` 文件。
- `pg_restore -l` 可读取目录清单。
- 备份文件复制到独立存储后校验 SHA-256，并记录备份时间、数据库、Migration 版本和校验值。

## 4. 隔离恢复流程

恢复前必须确认目标数据库为新建验证库。示例名称 `sim_platform_restore_verify_YYYYMMDD`。

```bash
docker compose exec -T postgres createdb -U sim sim_platform_restore_verify_20260622
docker compose exec -T postgres pg_restore -U sim -d sim_platform_restore_verify_20260622 --no-owner --no-privileges /backups/sim_platform_20260622_prechange.dump
```

恢复校验：

```bash
docker compose exec -T postgres psql -U sim -d sim_platform_restore_verify_20260622 -c "SELECT version_num FROM alembic_version;"
docker compose exec -T postgres psql -U sim -d sim_platform_restore_verify_20260622 -c "SELECT count(*) FROM config.site_maps;"
docker compose exec -T postgres psql -U sim -d sim_platform_restore_verify_20260622 -c "SELECT count(*) FROM message.message_records;"
```

校验通过后才允许安排切换：

1. 停止目标 Workspace 的 Platform API、Agent 和 Executor 写入。
2. 再次备份原运行库。
3. 将 Workspace `DATABASE_URL` 切换到已验证的恢复库，或按变更单执行数据库重命名。
4. 启动 Platform API，确认 Migration、健康检查、地图版本、消息数量和 Redis 预热。
5. 完成一条只读查询和一条受控 `where` 指令链路验证。

## 5. Redis 恢复

Redis Key 统一前缀：

```text
sim:workspace:{workspaceId}:runtime:robots
sim:workspace:{workspaceId}:runtime:messages
sim:workspace:{workspaceId}:events
```

Redis 数据不执行恢复。Redis 清空或丢失后，重启 Platform API 即从 PostgreSQL 预热机器人状态和最近消息；后续 MQTT 与 API 写入继续更新缓存。

## 6. 1～20 Workspace 扩展规则

- P5 实施后每 Workspace 使用独立 Database 和 Role，备份按 Workspace Database 分别生成。
- Control Plane Database 单独备份，禁止与 Workspace 业务库合并恢复。
- 恢复演练至少每季度执行一次，并随机抽取一个 Workspace 验证。
- Workspace 备份元数据由 Control Plane 记录；备份文件仍存放在受控对象存储或加密文件系统。

## 7. 禁止事项

- 禁止直接对运行库使用 `pg_restore --clean`。
- 禁止未验证备份清单就删除原数据库或 Volume。
- 禁止把 Redis Volume 当作主存储备份。
- 禁止在 Platform API 持续写入期间执行数据库替换。
- 禁止在未确认目标数据库名称时执行删除或覆盖操作。
