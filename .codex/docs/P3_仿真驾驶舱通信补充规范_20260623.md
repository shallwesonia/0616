## 1. 通信边界

P3 不改变机器狗 MQTT 基础契约：

```text
command topic: factory/dogs/{robotCode}/command
result topic:  factory/dogs/{robotCode}/result
```

仿真驾驶舱必须遵守：

- 前端不直连 MQTT Broker。
- 前端不直接调用虚拟机器人执行体。
- 前端不直接修改机器人状态。
- 前端所有动作必须先创建 `Action`。
- `Action` 通过平台消息总成转换为 MQTT command。
- MQTT result 必须先记录为 `MessageRecord`，再标准化为 `Observation`。
- `Observation` 聚合后更新 `CurrentState.stateVersion`。

## 2. 标准链路

```text
/simulation
-> POST /api/v1/actions
-> Action Service
-> MessageRecord(command)
-> MQTT Bridge publish command
-> factory/dogs/{robotCode}/command
-> virtual-robot-runner
-> factory/dogs/{robotCode}/result
-> MQTT Bridge receive result
-> MessageRecord(event)
-> Observation
-> CurrentState
-> TraceSpan
-> WebSocket simulation.snapshot
```

CurrentState 来源规则：

- 当外部 `scene-world-state-hub` 可用时，驾驶舱优先读取 `GET /api/v1/integrations/hub/current-state`。
- 接口返回的 `environmentState.source=hub` 表示展示侧使用 Hub 运行态事实源。
- Hub 不可用、当前 Scene 未注册或 CurrentState 不存在时，驾驶舱回退到 `GET /api/v1/current-states/{runId}`。
- 前端只展示来源标识和状态，不直接写入 Hub CurrentState。

## 4. 消息分类

| 标准分类 | 来源 | 说明 |
|---|---|---|
| Command | Platform API / MQTT Bridge | 平台下发的动作指令 |
| Ack | MQTT result | 指令接收、拒绝、开始处理等确认 |
| Telemetry | MQTT result | 位置、心跳、where 结果等遥测 |
| Event | MQTT result / 平台内部 | 任务开始、任务完成、快照、运行事件 |
| Alert | MQTT result / 异常注入 | 离线、失败、超时、阻塞、消息丢失 |

事件映射：

| event | category |
|---|---|
| `command.accepted` | Ack |
| `command.rejected` | Ack |
| `pose.updated` | Telemetry |
| `where.result` | Telemetry |
| `task.started` | Event |
| `task.succeeded` | Event |
| `task.failed` | Alert |
| `task.timeout` | Alert |
| `device.offline` | Alert |
| `robot.offline` | Alert |
| `action.failed` | Alert |
| `path.blocked` | Alert |
| `interface.timeout` | Alert |
| `message.dropped` | Alert |
| `resource.locked` | Alert |
| `station.unavailable` | Alert |
| `battery.low` | Alert |

## 5. 异常注入规范

异常注入入口：

```text
POST /api/v1/simulation-runs/{runId}/events
```

支持事件：

```text
robot.offline
action.failed
path.blocked
interface.timeout
message.dropped
resource.locked
station.unavailable
battery.low
```

处理规则：

- 异常注入创建 `MessageRecord(event)`。
- 随后创建 `Observation(category=Alert)`。
- `Observation` 更新 `CurrentState.activeEvents`。
- 路径阻塞类事件更新 `CurrentState.resourceStates.blockedPaths`。
- Trace 记录 `observation.{event}` span。
- 默认不向真实机器人网关发布异常控制指令。

## 6. 导出规范

单次 Run 导出入口：

```text
GET /api/v1/simulation-runs/{runId}/export
```

导出内容：

```text
manifest
workspace
run
tasks
actions
observations
currentState
snapshots
traces
mqttMessages
audit
```

导出包不得包含：

- MQTT 密码。
- JWT。
- 证书私钥。
- 数据库连接串。
- Docker secret。

## 7. 语义澄清：Action 停止、Sandbox 回放和离线事件

### 7.1 Action 停止语义

`POST /api/v1/actions/{actionId}/stop` 表示平台侧停止某条既有 Action 的生命周期管理：

- 更新目标 Action 状态为 `Stopped`。
- 写入 Trace，例如 `action.stopped`。
- 不默认创建新的 MQTT command。
- 不默认发布 `factory/dogs/{robotCode}/command`。
- 不等同于机器人已经物理停止。

`command=stop` 表示通过消息总成向机器人执行体下发停止指令：

- 必须创建新的 Action。
- 必须生成新的 `commandId`。
- 必须通过 MQTT Bridge 发布到 `factory/dogs/{robotCode}/command`。
- 不直接篡改历史 Action 状态。

如果业务需要同时“让机器人停止”和“结束原 Action”，应采用组合流程：

```text
创建 command=stop 的新 Action
-> 消息总成下发 MQTT stop command
-> 机器人返回 task.stopped / command.accepted 等 result
-> 平台根据 result 关闭或标记原 Action
```

### 7.2 Sandbox 回放语义

当前 `sandbox=true` 的消息回放不触发 MQTT 发布，也不驱动机器人执行体重新执行。

但当前实现仍会写入原 Workspace 的调试数据，包括：

- `MessageRecord`
- `Observation`
- `Trace`
- 可能更新 `CurrentState.stateVersion`

因此在未引入独立 `ReplayWorkspace` 或真正 `dryRun=true` 前，前端应禁用或显著标记 sandbox 回放入口。

### 7.3 robot.offline 语义

仿真驾驶舱中的 `robot.offline` 是上层主动注入的模拟故障事件，用于验证任务流、异常处理和智能体调度策略。

正式机器人环境中，机器人离线事实必须由设备端、机器人网关、心跳监控或平台检测产生，建议使用：

```text
device.offline
heartbeat.timeout
robot.gateway.offline
```

统一分类仍为 `Alert`，但事件来源必须清晰区分。
