## 1. 页面定位

仿真驾驶舱是独立于地图编辑页的运行工作台。

职责：

- 创建和启动 SimulationRun。
- 创建 Task 并观察执行状态。
- 查看 Plan 版本、步骤和重规划原因。
- 查看 Action、MQTT command 和 result event。
- 查看 Observation、CurrentState、Snapshot 和 Trace。
- 在授权模式下触发控制台事件和手动联调 Action。
- 导出单次 Run 的完整联调包。

非职责：

- 不编辑地图正式版本。
- 不直接修改机器人状态。
- 不直接连接 MQTT command Topic。
- 不在前端执行任务规划和 AI 决策。
- 不替代 Platform API、Agent Service 或 Robot Executor。

## 2. 核心模型

建议增加 `SimulationRun` 作为一次运行容器：

```text
Workspace
  └─ SimulationRun
      └─ Task
          └─ Plan v1/v2...
              └─ Action
                  └─ MQTT command
                      └─ MQTT result
                          └─ Observation
                              └─ CurrentState
                                  └─ Snapshot

Trace 贯穿全部对象
```

所有对象必须包含 `workspaceId` 和 `runId`。

## 3. 对象职责

| 对象 | 定义 | 禁止混入的内容 |
|---|---|---|
| Task | 业务目标和完成条件 | 具体设备指令序列 |
| Plan | 实现 Task 的可版本化步骤 | 真实执行结果 |
| Action | 一次可执行尝试 | 多个无关设备动作 |
| Observation | 外部或内部已发生事实 | 下一步决策 |
| CurrentState | Observation 聚合后的当前投影 | 历史日志全集 |
| Snapshot | 指定状态版本的不可变冻结 | 高频实时状态流 |
| Trace | 全链路关联和时间线 | 原始业务 Payload 副本 |

## 4. 标识标准

```text
workspaceId
runId
taskId
planId
planVersion
planStepId
actionId
commandId
requestId
observationId
stateVersion
snapshotId
traceId
spanId
parentSpanId
```

映射规则：

- `taskId`、`commandId`、`requestId`、`traceId` 与机器狗 MQTT 协议保持一致。
- `actionId` 是平台内部执行对象，通常与一个 `commandId` 对应。
- `stop` 创建新 `commandId`，复用目标 Task 的 `taskId` 和 `traceId`。
- `where` 使用 `requestId`；不属于业务任务时 MQTT `taskId=null`。

## 5. 状态机

Task：

```text
Draft → Ready → Running → Succeeded
                    ├→ Paused → Running
                    ├→ Failed
                    └→ Cancelled
```

Plan：

```text
Generating → Ready → Active → Completed
                         ├→ Superseded
                         └→ Failed
```

Action：

```text
Pending → Issued → Accepted → Running → Succeeded
                    ├→ Rejected
                    ├→ Failed
                    ├→ Stopped
                    └→ Timeout
```

Observation：

```text
Received → Validated → Applied
              ├→ Ignored
              └→ Invalid
```

Trace：

```text
Open → Completed
   └→ Error
```

## 6. 对象标准字段

### 6.1 Task

```text
taskId、workspaceId、runId、traceId、goal、input、constraints、priority、
deadline、expectedOutcome、status、createdBy、createdAt、startedAt、finishedAt
```

### 6.2 Plan

```text
planId、taskId、traceId、planVersion、strategy、steps、dependencies、
assumptions、generatedBy、generationLatencyMs、status、createdAt、activatedAt
```

Plan Step：

```text
planStepId、sequence、actionType、target、params、dependsOn、
successCondition、failurePolicy、timeoutMs
```

### 6.3 Action

```text
actionId、planId、planStepId、taskId、traceId、robotCode、command、params、
commandId、requestId、attemptNo、timeoutMs、status、issuedAt、startedAt、finishedAt
```

只有 Action Service 可以将 Action 转换成 MQTT command。

### 6.4 Observation

```text
observationId、workspaceId、runId、taskId、actionId、traceId、source、event、
eventId、robotCode、commandId、requestId、timestamp、data、error、processingStatus
```

Observation 来源：

- MQTT result event。
- 控制台事件。
- 资源状态变化。
- Agent 内部计时器。
- API 集成事件。
- 人工确认。

### 6.5 CurrentState

```text
workspaceId、runId、stateVersion、taskState、activePlan、robotStates、
resourceStates、environmentState、pendingActions、activeEvents、
lastObservationId、lastObservationAt、updatedAt
```

CurrentState 是可重建的派生数据，不是历史事实源。

### 6.6 Snapshot

```text
snapshotId、workspaceId、runId、taskId、traceId、stateVersion、reason、
task、activePlan、actionCursor、robotStates、resourceStates、environmentState、
pendingActions、createdAt、checksum
```

Snapshot 触发时机：

- Run 启动。
- Task 启动和结束。
- Plan 激活或重规划。
- Task 暂停或恢复。
- 严重异常发生。
- 人工创建检查点。
- Run 结束。

禁止按每秒固定频率生成完整 Snapshot。

### 6.7 Trace

Trace Header：

```text
traceId、workspaceId、runId、taskId、status、startedAt、finishedAt、durationMs
```

Trace Span：

```text
spanId、parentSpanId、traceId、entityType、entityId、operation、status、
startedAt、finishedAt、durationMs、inputRef、outputRef、errorRef
```

Trace 只保存引用和关键摘要，原始 MQTT Payload 继续保存在 MessageRecord。

## 7. 标准运行闭环

```text
创建 SimulationRun
→ 绑定地图配置版本和机器人
→ 创建 Task
→ 生成初始 Snapshot
→ Agent 读取 CurrentState
→ 生成 Plan v1
→ 激活 Plan
→ 创建 Action
→ Platform API 校验 Action
→ 发布 MQTT command
→ 机器狗返回 result event
→ Message Hub 创建 Observation
→ Observation 更新 CurrentState
→ Agent 判断继续、重试、停止或生成 Plan v2
→ Task 完成
→ 创建最终 Snapshot
→ 关闭 Trace
```

随机事件和控制台事件必须先创建 Observation，再影响 CurrentState 和 Plan。

## 8. 驾驶舱模式

| 模式 | 用途 | 控制边界 |
|---|---|---|
| 观察模式 | 只看运行状态和 Trace | 不允许创建 Task 或 Action |
| 手动联调模式 | 手工发起 goto_pose、where、stop | 通过 Action API，不直连 MQTT |
| 任务流模式 | 按预定义 Task 模板执行 | Agent 使用规则策略生成 Plan |
| 智能体模式 | AI 或规则智能体动态规划 | Action 必须经过平台校验 |
| 回放模式 | 从 Snapshot 和 Observation 重建过程 | 不发布真实 MQTT command |

```text

POST /api/v1/simulation-runs/{runId}/tasks/from-template
POST /api/v1/simulation-runs/{runId}/tasks/batch

```

驾驶舱建任务模式：

- 手动任务：创建单个 `Task`，自动生成默认 `Plan v1`。
- 模板任务：从 `TaskTemplate` 生成单个 `Task`。
- 批量任务：一次生成多个相互独立的 `Task`，适合压力和随机场景。
- 连续任务：创建 `TaskChain`，将连续业务拆成多个标准 `Task`，每个 `Task` 独立持有自己的 `Plan`、`Action`、`Trace`。

Plan 建立规则：

- 默认创建 Task 时自动生成 `Plan v1`。
- 驾驶舱可通过 `POST /api/v1/tasks/{taskId}/plans` 创建手动 `Plan vNext`。
- `activate=true` 时，新 Plan 成为 activePlan，旧 activePlan 标记为 `Superseded`。
- Plan 重规划不覆盖历史版本。

Action 与 Observation：

```text

GET  /api/v1/robots
POST /api/v1/actions

GET  /api/v1/actions/{actionId}
POST /api/v1/actions/{actionId}/stop
GET  /api/v1/observations
GET  /api/v1/current-states/{runId}
```

Snapshot 与 Trace：


## 14. Workspace 隔离

- 驾驶舱只能访问当前 JWT `workspaceId`。
- 页面路由、API、WebSocket、数据库和 MQTT Broker 均属于当前 Workspace。
- 一个 Workspace 的 Snapshot、Trace、日志和导出文件不得被另一个 Workspace 查询。
- 用户切换 Workspace 时必须重新加载连接信息和运行上下文。
- 前端不得缓存其他 Workspace 的 CurrentState。

## 15. 数据一致性

- Observation 和 MessageRecord 采用追加写入。
- 使用 `eventId/messageId` 去重。
- Action 使用 `commandId` 幂等。
- CurrentState 按 `stateVersion` 乐观更新。
- 同机器人 Observation 按设备时间和平台接收时间处理乱序。
- 延迟 result 必须记录，但不得覆盖较新的 CurrentState。
- Snapshot 必须引用明确的 `stateVersion`。
- Plan 重规划必须创建新版本，旧版本不可覆盖。

## 16. 指标

- Task 完成率和平均耗时。
- Plan 重规划次数。
- Action 成功率、拒绝率、超时率和重试次数。
- Observation 延迟和无效消息数。
- CurrentState 更新延迟。
- Snapshot 数量和存储量。
- Trace 完整率。
- 机器人利用率、资源等待时间和异常次数。

## 19. 验收标准

- 驾驶舱与地图编辑页面职责分离。
- 用户只能访问授权 Workspace 的驾驶舱。
- Task、Plan、Action、Observation、CurrentState、Snapshot、Trace 可以完整串联。
- Action 可追踪到 MQTT command 和全部 result event。
- result event 可生成 Observation 并更新 CurrentState。
- Plan 重规划产生新版本且旧版本可查询。
- Snapshot 可恢复指定检查点的只读状态。
- Trace 可说明任务为何成功、失败、等待或重规划。
- WebSocket 断开后可回退 REST 查询。
- 单次 Run 可导出完整且不包含 Secret 的联调包。

## 20. 手动编排任务补充：2026-06-30

本阶段在驾驶舱建任务区新增“编排”模式，采用 MVP 落地方式：

- 一个业务目标对应一个 `Task`。
- 任务创建后通过 `POST /api/v1/tasks/{taskId}/plans` 创建 `strategy=manual_orchestration` 的手动多步 `Plan vNext`。
- `PlanStep` 采用串行依赖，默认后一环节 `dependsOn` 前一环节。
- 当前支持编排动作：`goto_pose`、`pick`、`place`、`wait`、`where`。
- 支持 A-B 往返循环快捷生成，但循环在前端展开为普通 `PlanStep`，不引入嵌套 `LoopStep`。
- 编排创建完成后不直接下发机器人指令；后续仍由 Action / Command / 消息总成链路逐步执行。

典型编排：

```text
1. goto_pose 到取货点
2. pick 抓取目标
3. goto_pose 到放置点
4. place 放置目标
5. goto_pose 返回待命点
```

边界：

- 本阶段不实现并行步骤。
- 本阶段不实现条件分支。
- 本阶段不实现无限循环。
- 本阶段不自动推进 PlanStep 到 Action，后续由规则调度器或 Agent Service 接入。

## 22. 语义澄清补充：2026-06-23

### 22.1 Action 停止方式

驾驶舱中存在两类停止语义：

- `POST /api/v1/actions/{actionId}/stop`：平台侧停止既有 Action 的生命周期管理，不默认发布 MQTT stop。
- `command=stop`：通过消息总成创建新的 stop Action，并向机器人 command Topic 下发停止指令。

驾驶舱页面文案和后端测试必须避免将二者混用。

### 22.2 Sandbox 回放入口

当前 sandbox 回放不触发 MQTT，也不驱动机器人执行体，但会写入当前 Workspace 的 MessageRecord、Observation、Trace，并可能影响 CurrentState 版本。

在真正隔离的 ReplayWorkspace 或 dry-run 机制完成前，驾驶舱应暂时禁用或显著提示 sandbox 回放入口。

### 22.3 robot.offline 事件语义

`robot.offline` 在驾驶舱中仅表示仿真控制台注入的模拟故障。

正式机器人环境中的离线事实应由设备端、机器人网关或心跳监控上报，建议使用 `device.offline`、`heartbeat.timeout` 或 `robot.gateway.offline`。

## 23. 多机器人实施补充：2026-06-24

本次将驾驶舱和运行模型从单机器人扩展为默认 3 机器人编队。

已实现内容：

- 默认场景包含 `robot-001`、`robot-002`、`robot-003`。
- Docker Compose 默认启动 3 个独立虚拟执行体服务，每个执行体使用独立 `ROBOT_CODE` 和 MQTT clientId。
- `ScenarioSummary` 增加机器人明细，包含 `robotCode`、`robotType`、初始坐标、状态和能力摘要。
- 创建 Action 时按 `robotCode` 绑定目标机器人；未知 `robotCode` 直接拒绝。
- 后端按 MQTT result 中的 `robotCode` 聚合机器人状态，`CurrentState.robotStates` 返回全部机器人状态数组。
- 驾驶舱在“选场景”区域提供新增机器人入口，可填写 `robotCode`、机器人类型和初始坐标。
- 驾驶舱支持多机器人状态卡片展示，点击机器人后同步切换发指令目标。
- Action 队列显示 `robotCode / command`，用于识别每条 Action 的目标执行体。
- 机器人类异常注入使用机器人下拉选择，支持对单台机器人注入离线、动作失败等异常。

边界保持：

- 前端仍不直连 MQTT，也不直接调用虚拟执行体。
- 新增机器人入口只登记平台配置和初始状态，不自动启动虚拟执行体容器，也不向 MQTT 发布控制指令。
- 多机器人能力只解决状态展示、单机器人指令路由和单机器人异常注入，不包含多机器人调度优化算法。
- 真实机器人替换时，只需将对应虚拟执行体服务停用，并让真实机器人网关复用相同 `robotCode`、command/result Topic 和 Payload 契约。

验收要求：

- 三台虚拟机器人可同时在线。
- 任意一台机器人可独立接收 Action command。
- 任意一台机器人可独立上报 `pose.updated`、`action.*`、`task.*`、`device.offline` 等 result event。
- 驾驶舱可同时展示三台机器人状态、位置、进度和当前动作。
- 控制台可对单台机器人注入异常，不影响其他机器人继续在线。
